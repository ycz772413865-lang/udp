=============================================
              Task2: UDP Socket
=============================================

运行环境：
- Python 3.8+
- 需要安装 pandas 库（用于UDP数据统计）
- 需要安装 portalocker 库（用于文件锁，防止日志写入竞争）

安装依赖：
  pip install pandas portalocker

=============================================
              文件列表
=============================================

- udp_server.py  : UDP服务端程序
- udp_client.py  : UDP客户端程序
- test.txt       : 测试文本文件（需提前准备）
- run_log.txt    : 服务端和客户端共用运行日志（自动生成）

=============================================
              运行步骤
=============================================

1. 启动UDP服务端：
   python udp_server.py

   默认监听端口：12346
   丢包率：30%（可在代码中修改 drop_rate 变量）

2. 启动UDP客户端：
   python udp_client.py <server_ip> <server_port>

   参数说明：
   - server_ip   : 服务端IP地址（如 127.0.0.1）
   - server_port : 服务端端口（如 12346）

   示例：
   python udp_client.py 127.0.0.1 12346

=============================================
              功能说明
=============================================

总体设计：
  基于UDP协议，在应用层模拟TCP的可靠传输机制，实现文件数据的可靠传输。

服务端（udp_server.py）：
- 创建UDP Socket监听指定端口
- 解析报文头部的序列号（seq_num）和类型（type_field）
- 支持多客户端独立状态管理（每个客户端维护独立的期望序列号）
- 随机丢包模拟网络不可靠性（丢包率可配置）
- StudentID 校验机制（异或加密验证）

客户端（udp_client.py）：
  UDPClient 类实现完整的可靠传输协议：

  1. 连接建立（模拟TCP三次握手）
     - 客户端发送 SYN 报文（含 StudentID 异或加密）
     - 服务端回复 SYN-ACK 报文（含服务器时间）
     - 客户端发送 ACK 确认，连接建立
     - 支持超时重传（默认最多重试5次）

  2. 数据传输（基于GBN协议，Go-Back-N）
     - 滑动窗口大小: 5
     - 每个数据包大小: 80 字节
     - 超时时间: 300ms
     - 发送窗口内的数据包，等待 ACK
     - 超时未收到 ACK 则重传窗口内所有未确认包
     - 服务端仅按序接收，乱序包触发重发期望的 ACK

  3. 连接断开（模拟TCP四次挥手）
     - 客户端发送 FIN 报文
     - 服务端回复 FIN-ACK 报文
     - 支持超时重传（默认最多重试5次）

  4. 数据统计（使用 pandas 计算）
     - 丢包率
     - 实际发送报文总数
     - RTT 最大值、最小值、平均值、标准差
     - 服务端时间戳显示

=============================================
              报文格式
=============================================

固定头部格式（至少8字节）：

  - SeqNum (4字节, big-endian) : 序列号
  - Type (4字节, big-endian) : 报文类型
  - Payload (变长) : 根据类型不同

报文类型：
  - Type=1: SYN（客户端→服务端）
           负载: 2字节 StudentID（学号 ^ 0x5A3C 异或加密）
  - Type=2: SYN-ACK（服务端→客户端）
           负载: 8字节 服务器时间（HH-MM-SS ASCII编码）
  - Type=3: 数据报文（客户端→服务端）
           负载: 文件数据（最多80字节/包）
  - Type=4: ACK（服务端→客户端）
           负载: 8字节 服务器时间（HH-MM-SS ASCII编码，可选）
  - Type=5: FIN（客户端→服务端）
           负载: 无
  - Type=6: FIN-ACK（服务端→客户端）
           负载: 8字节 服务器时间（HH-MM-SS ASCII编码）

=============================================
              协议细节
=============================================

连接建立：
  1. 客户端生成 SYN 报文：seq=0, type=1
     StudentID = (学号) ^ 0x5A3C
  2. 服务端收到后校验 StudentID，解码后应为有效学号（0-9999）
  3. 服务端回复 SYN-ACK：seq=0, type=2，携带服务器时间
  4. 客户端回复 ACK：seq=1, type=4
  5. 连接建立成功

数据传输（GBN协议）：
  - 客户端维护 base（窗口基序）和 next_seq（下一个待发送序）
  - 窗口范围: [base, base + WINDOW_SIZE)
  - 发送窗口内所有可发送的数据包
  - 等待 ACK，收到后推进 base
  - 超时 (300ms) 则重传 [base, next_seq) 范围内所有未确认包
  - 服务端维护每个客户端的 expected_seq，仅按序接收

连接断开：
  1. 客户端停止接收线程，发送 FIN（type=5）
  2. 服务端回复 FIN-ACK（type=6）
  3. 服务端清除该客户端状态
  4. 连接断开

=============================================
              数据统计
=============================================

程序运行结束后输出以下统计信息：
  - 丢包率（%）= (实际发送总数 - 数据包总数) / 实际发送总数 × 100
  - 实际发送报文总数（含重传）
  - RTT 最大值（ms）
  - RTT 最小值（ms）
  - 平均 RTT（ms）
  - RTT 标准差（ms）

=============================================
              输出文件
=============================================

运行结束后会自动生成以下日志文件：

- run_log.txt：服务端和客户端共用运行日志
  使用 portalocker 文件锁防止多进程写入竞争
  日志格式：时间戳 [UDP_SERVER/UDP_CLIENT] 事件内容
  记录服务端和客户端收发每一类报文（SYN/SYN-ACK/DATA/ACK/FIN/FIN-ACK）
  包含序列号、时间戳、丢包标记、重传记录和地址信息

=============================================
              注意事项
=============================================

1. 运行前确保 test.txt 文件存在于程序目录下
2. UDP服务端和客户端使用端口 12346（与TCP的12345不同）
3. 服务端丢包率默认 30%，可在代码中修改 drop_rate 变量（0.0-1.0）
4. 客户端 StudentID 硬编码为 2111（可在代码中修改 student_number 变量）
5. 建议使用 Wireshark 抓取网络报文进行分析
6. 修改服务器IP和端口时，确保两端配置一致
7. 需安装 pandas 和 portalocker 库：pip install pandas portalocker
